A systematic investigation of positive-unlabeled learning on code representations

Introduction

Software vulnerability detection is typically framed as a binary classification problem. Given a function's source code, predict whether it contains a security vulnerability. This formulation assumes reliable positive and negative labels. In practice, that assumption falls apart.

Functions confirmed as vulnerable, through CVE reports, security advisories, and patch commits, are trustworthy positives. But "benign" labels are assigned by exclusion. A function is called safe simply because nobody has found a vulnerability in it yet.

A note on security terminology
Two acronyms come up repeatedly in vulnerability detection and they are easy to conflate. A CWE (Common Weakness Enumeration) is a category of bug. CWE-119, for example, covers buffer overflows; CWE-362 covers race conditions. Think of CWEs as the taxonomy. A CVE (Common Vulnerabilities and Exposures) is a specific, publicly disclosed instance of a bug in a particular piece of software. CVE-2014-0160 is the Heartbleed bug. It is one instance of CWE-119. In the PrimeVul dataset, every vulnerable function has an associated CVE (the specific disclosure) and one or more CWEs (the type of flaw). When I later analyze performance "by CWE type," I am grouping vulnerabilities by the nature of the bug, not by individual disclosures.

In real-world application security workflows, the process for generating labels looks like this: a security team runs static analysis tools against the codebase written by their internal engineering teams. These tools flag code patterns that match known CWE categories. The tools are intentionally aggressive and produce a lot of false positives. An application security expert then triages each finding, reads the surrounding code, traces data flows, and determines whether the flagged pattern is actually exploitable in context. This process takes 15–60 minutes per finding.

The result is the PU structure shown in Figure 4: a small set of confirmed positives and a large pool of unlabeled data with no confirmed negatives. PU learning has been studied for vision and tabular data [12], and PILOT [4] recently demonstrated it can work for vulnerability detection. But a fundamental question remains unanswered:

Hypothesis
There exists a minimum number of labeled positives below which PU learning collapses, and above which it recovers most of the performance of full supervision. The two dominant families of PU methods, risk estimators and pseudo-labeling, hit this threshold at different points and for different reasons. Effectiveness also varies across vulnerability types: syntactically distinctive bugs should be easier to detect with limited labels than semantically subtle ones.

This project uses vulnerability detection as a laboratory for studying PU learning behavior. Rather than proposing a new method, I investigate when PU learning works on code, how many labeled examples it actually needs, and what makes it break.

Background

The Labeling Problem

Consider a security team responsible for a large C/C++ codebase. They run static analysis tools that flag thousands of potential issues. Each one requires manual review: an expert reads the surrounding code, traces data flows, and determines whether the pattern constitutes a real, exploitable vulnerability. This process takes 15–60 minutes per finding. The team labels the clear cases, confirmed vulnerabilities and obvious false positives, but the ambiguous middle ground (the majority) remains unreviewed.

The result is a dataset with two kinds of labels:

Labeled positives (P) are functions with confirmed vulnerabilities, backed by CVE reports and security patches. These labels are reliable. Unlabeled data (U) is everything else, a collection of functions that might be safe or might contain undiscovered flaws. There are no confirmed negatives. No function has been certified as "definitely not vulnerable."

PU learning: two families

PU learning addresses exactly this setting. Train a binary classifier given only P and U, without assuming U is purely negative. Two distinct families of methods have emerged [12], [5]:

Key Distinction
Risk estimator methods modify the classification loss to mathematically account for hidden positives in U. Pseudo-labeling methods iteratively identify likely negatives in U, then train a standard classifier on the expanded label set. These two families make different assumptions and fail in different ways.

Risk estimator methods. The foundational unbiased PU risk estimator [1] rewrites the binary classification risk using only the P and U distributions, plus an estimate of the class prior (the true fraction of positives). This is elegant but fragile: the empirical risk can go negative, destabilizing training. Non-negative PU (nnPU) [2] clips the negative risk term. GradPU [3] adds a gradient penalty for Lipschitz continuity. These methods all require knowing or estimating the class prior, a quantity that is rarely available in practice. I use nnPU as the representative of this family. It is the standard baseline that all later methods compare against, and simple enough to isolate the approach's behavior from implementation complexity.

Pseudo-labeling methods. Self-training [11] takes a simpler approach. Train an initial classifier treating U as negative, use it to score U and select high-confidence negatives, retrain on P plus selected negatives, and repeat. PILOT [4] extends this with distance-aware label selection for vulnerability detection. These methods are prior-free. They don't need an estimate of the class prior, but they depend on the quality of the initial representations.

A NeurIPS 2025 empirical study [5] compared these two families head-to-head on standard benchmarks and proposed a unified framework. However, their evaluation used only general-purpose datasets, not code representations or domain-specific tasks with extreme class imbalance.

Vulnerability Detection with Code Models

Modern vulnerability detection relies on pretrained code language models. Devign [8] uses graph neural networks over code property graphs. LineVul [9] showed strong results by fine-tuning CodeBERT [7] for vulnerability classification. Chakraborty et al. [10] highlighted that label noise is a serious problem in vulnerability datasets, with "benign" labels being particularly unreliable, which is exactly the motivation for PU learning. PrimeVul [6] addresses these label quality issues and provides cleaner benchmarks with chronological splits and deployment-oriented metrics.

The Gap

PILOT [4] showed PU learning is viable for vulnerability detection, but focused on proposing a new method rather than investigating when and why PU learning works. The NeurIPS 2025 study [5] compared PU approaches but only on general benchmarks, not code. No prior work evaluates: (a) how PU methods perform as a function of labeling budget on code representations, (b) how sensitive risk estimators are to class prior estimation at extreme imbalance (~3% positive), or (c) how performance varies across vulnerability categories with different syntactic signatures.

Approach

Why Not End-to-End Fine-Tuning?

The natural approach would be to fine-tune CodeBERT end-to-end for each experimental configuration: each PU method, each labeling fraction, each class prior value, each random seed. With 4 methods, 7 labeling fractions, 5 prior values, and 3 seeds, that is over 100 training runs. Each fine-tuning run on 176K functions takes 5–10 hours on a single GPU. I estimated the total compute cost and:

Instead, I use a feature extraction strategy that decouples the representation from the PU investigation. The pipeline has two stages. First, every function is passed through a frozen CodeBERT model [7] exactly once, producing a 768-dimensional [CLS] embedding that serves as the function's fixed representation. Second, all PU experiments operate on these embeddings with lightweight MLP classifiers that train in seconds on CPU. This cuts a 500+ GPU-hour experiment plan down to about an hour of CPU time, which is what lets me produce proper error bars and sweep across dozens of configurations.

The pipeline

The key property of this design is that Stages 2 and 3 are cheap. Changing the PU method, the labeling fraction, the class prior, or the random seed means retraining a 197K-parameter MLP on precomputed vectors. No GPU required. This is what makes the sweeps across dozens of configurations practical for a single person on a laptop.

I compare four methods spanning two PU families plus bounds:

Experimental setup

Dataset and PU simulation

I use PrimeVul v0.1 [6], which contains 175,797 training functions (4,862 vulnerable, 170,935 benign), with 23,948 for validation and 24,788 for testing. The splits are chronological: training data comes from older commits, test data from newer ones. PrimeVul also provides 435 matched pairs of vulnerable functions and their patched versions for pairwise evaluation.

To simulate the PU setting, I sample a fraction of the vulnerable training functions as the labeled positive set P. The rest, the remaining vulnerable functions plus all benign functions, becomes U. I strip their labels. At 20% labeling (the default), P contains 972 functions and U contains 174,825 functions, of which 3,890 are hidden vulnerabilities that the model does not know about. The validation and test sets keep their full labels throughout.

Representation

I tokenize each function's source code and pass it through a frozen CodeBERT model [7] (microsoft/codebert-base, 125M parameters). I extract the [CLS] token embedding from the final layer, producing a 768-dimensional vector per function. This is done once for all splits.

Classifier

All methods use the same architecture: a 2-layer MLP with a 256-unit hidden layer, ReLU activation, and 30% dropout. The model takes a 768-d embedding and outputs a single logit (sigmoid applied at inference). That is 197,377 parameters. I train with Adam (lr=1e-3) and early stopping on validation F1 score with patience of 5-7 epochs depending on the method.

The supervised and naive PU baselines use BCE loss with a positive class weight (n_neg / n_pos) to handle the class imbalance. For nnPU, the loss is the PU risk estimator from Kiryo et al. [2], which approximates the binary classification risk from P and U alone, given a class prior estimate. When the negative risk term goes below zero (a known instability), nnPU clips it. For self-training, I run 5 iterations. Each iteration after the first uses the current model to score U and selects the 10% of U with the lowest predicted vulnerability probability as "reliable negatives," then retrains a fresh MLP on P plus these selected negatives.

Experiments

I run three experiments:

1. Baseline comparisonMethod (all 4)20% labeled (P=972), prior=0.0277, 3 seedsWhich method works best? How close to supervised?
2. Labeling budgetLabeled fraction: 2%-80% (P=97 to P=3,890)nnPU + self-training, 3 seedsHow many labeled positives does PU learning need? Is there a cliff?

Experiment 2 is the central one. The percentage (2%, 5%, 10%, etc.) is specific to PrimeVul's 4,862 vulnerable functions. The more generalizable quantity is the absolute P count. At 2% labeling, P contains 97 functions. At 5%, 243. At 10%, 486. The labeling curve will show whether there is a smooth degradation or a cliff below which PU learning stops working, and whether nnPU and self-training hit that threshold at different points. I plot both the percentage and the absolute count on the x-axis so the result translates to other datasets. A security team deciding how many findings to label cares about the absolute number, not the fraction of a denominator they do not know.

I also stratify test set results by CWE type (for categories with at least 10 test samples) to see whether syntactically distinctive vulnerability types like buffer overflows (CWE-119) hold up better under PU learning than semantically subtle ones like race conditions (CWE-362). All experiments use 3 random seeds (42, 123, 456) and I report mean and standard deviation.
